# Investigative Journalism & OSINT Methodology + Training: Offline-First Feature Map for Greyline

_Part of the Greyline research corpus. Generated from cited 2026 web research; see Sources._

## Executive summary

Greyline already implements the exact engineering pattern that the strongest journalism and OSINT tools converge on: pure client-side compute, copy-ready query generation that is never transmitted to a target, best-effort binary parsing in TypeScript, opt-in caching connectors gated behind explicit toggles, and bundleable datasets. That existing architecture makes an investigative-journalism pillar a natural and defensible extension of the product rather than a strategic pivot. The same primitives that power the EXIF stripper, the vault, the surveillance-camera Overpass connector, the MapLibre map, and the self-doxxing dork generator are the precise building blocks the leading OSINT organizations assemble into their flagship tooling.

The leading organizations split into a small number of reusable archetypes. The first is the **local document-analysis engine**: ICIJ Datashare (AGPL-3.0, self-hosted, explicitly runnable offline with "strictly no data" leaving the machine, built on Apache Tika + Tesseract OCR + CoreNLP/spaCy NER + Elasticsearch) and OCCRP Aleph (spaCy NER plus cross-dataset cross-referencing as its standout capability). Greyline can ship a far lighter, fully-local variant: drop documents into a case folder, run OCR + named-entity recognition + entity cross-referencing in-browser or in-Node, and index everything in the SQLite store the app already uses. Greyline's investigations are single-user, not 11.9M-record leaks, so the heavyweight server stack is unnecessary.

The second archetype is **defensive source and document hygiene**: Micah Lee's Dangerzone (now stewarded by Freedom of the Press Foundation, AGPL-3.0, 5.5k stars, v0.10.0 shipped 2 Dec 2025) rebuilds untrusted PDFs, Office files, and images from raw pixel data inside a network-disabled gVisor sandbox; SecureDrop versus GlobaLeaks define the source-submission threat model that superseded WikiLeaks; and EFF/CPJ/FPF publish concrete, current (2025-2026) border-crossing device checklists. Greyline already ships the EXIF stripper, the vault, and a border-prep tool, so these become an integrated "document and source protection" suite rather than green-field work.

The third archetype is **geolocation and chronolocation**, anchored by Bellingcat's toolkit (rebuilt in 2024 by staffer Johanna Wild). It centers on SunCalc (mourner/suncalc, BSD, zero dependencies) for shadow-based chronolocation via `tan(sun_altitude) = object_height / shadow_length`, plus a custom OSM feature-cluster geolocation search that is effectively a simplified Overpass front-end. Because Greyline already runs MapLibre and an Overpass connector, a sun/shadow calculator and a "find a place near a school and a supermarket and a sidewalk-street" query builder are incremental additions. Verification methodology is dominated by Caulfield's SIFT (Stop, Investigate the source, Find better coverage, Trace to original) and Tactical Tech's Exposing the Invisible Kit (evidence, verification, documentation, safety, plus an explicit chain-of-custody definition); both map onto Greyline's established checklist pattern and a case-file with cryptographic chain-of-custody hashing. Offline reverse-image work is solvable with perceptual hashing (blockhash-core pure-JS, or PDQ via WASM) for near-duplicate and re-use detection with no upload. Every proposed feature stays offline-first; only the optional reverse-image-search hand-off and satellite/Overpass tiles touch the network, and both fit Greyline's existing opt-in connector model.

## Key findings

### Local document-analysis engines (Datashare, Aleph)

**ICIJ Datashare is explicitly designed for offline/local operation, and shares Greyline's license.** ICIJ states: "You use it locally, on your computer, and you can even use it offline if necessary. Strictly no data comes out of Datashare, even to ICIJ." It is AGPL-3.0 (the same license as Greyline), actively maintained (v21.6.0, May 2026, roughly 735 stars), and its NER/OCR stack is Apache Tika + Tesseract + CoreNLP/OpenNLP + Elasticsearch. Greyline can replicate the *core* of this — OCR, named-entity extraction, and cross-referencing — far more lightly in SQLite, precisely because Greyline's investigations are single-user rather than 11.9M-record leaks. (https://github.com/ICIJ/datashare, https://www.icij.org/inside-icij/2019/11/what-is-datashare-frequently-asked-questions-about-our-document-analysis-software/)

**OCCRP Aleph's defining capability is cross-referencing across datasets.** Aleph uses spaCy language models plus fastText language identification (176 languages) to extract people, companies, and countries alongside phone numbers, emails, and IBANs, then cross-references those entities across datasets — the relationship-mapping that distinguishes it from a simple search index. This is the behavior Greyline should emulate locally: surface entities that recur across multiple documents in a case and visualize their co-occurrence. (https://docs.aleph.occrp.org/developers/explanation/entity-extraction/)

### Defensive document and source hygiene (Dangerzone, SecureDrop/GlobaLeaks, border security)

**Dangerzone's pixel-rebuild sanitization concept is portable to pure web tech with no container runtime.** Dangerzone (Micah Lee, now Freedom of the Press Foundation, AGPL-3.0, 5.5k stars, v0.10.0 released 2 Dec 2025) sanitizes untrusted documents by converting them to raw RGB pixel data inside a network-disabled gVisor sandbox, then rebuilding a fresh PDF outside — "almost like printing the document and then scanning it back in." It can optionally re-add a searchable text layer via OCR. The Include Security 2023 audit found no high-risk issues. The pixel-rebuild concept is implementable in pure web tech (pdf.js render-to-canvas, then re-encode, plus Tesseract.js OCR) without Docker, giving Greyline a no-container sanitizer that matches its "no external runtime" ethos. (https://github.com/freedomofpress/dangerzone, https://dangerzone.rocks/about/)

**SecureDrop and GlobaLeaks define the post-WikiLeaks source-submission landscape, but hosting is out of scope for a local-only app.** SecureDrop (FPF, Tor-native, used by the Guardian, NYT, the Washington Post, and The Intercept) relies on air-gapped/Qubes viewing of submissions and disk encryption; GlobaLeaks targets broader NGO use with Tor plus clearnet and easier deployment. Both are successors to WikiLeaks. Greyline can *teach* these concepts (not host them) via an explainer and a "before you receive a leak" checklist, since standing up a submission system is outside a local-only tool's remit. (https://www.cjr.org/tow_center_reports/guide_to_securedrop.php)

**EFF and FPF published a concrete, current journalist border-device checklist with citable operational facts.** EFF's June 2025 checklist and the "Digital Security 101: Crossing the US-Mexico Border" curriculum module (EFF's Dave Maass and FPF's Martin Shelton, with UTEP) establish key operational facts: border officers must put devices in airplane mode and cannot access live cloud content; powering a device fully off enables disk encryption and resists forensic analysis; prefer a PIN over biometric unlock. In FY2024 fewer than 0.01% of travelers had devices searched, reportedly rising in 2025. This extends Greyline's existing border-device-prep tool with citable, current guidance. CPJ's Digital Safety Kit adds practitioner steps: inventory device contents, back up and wipe, carry clean travel devices, enable remote wipe, and use a PIN rather than biometrics. (https://www.eff.org/deeplinks/2025/06/journalist-security-checklist-preparing-devices-travel-through-us-border, https://cpj.org/2019/07/digital-safety-kit-journalists/)

### Geolocation and chronolocation (Bellingcat toolkit, SunCalc, OSM search)

**SunCalc is the canonical chronolocation tool in Bellingcat's toolkit, and it is a hypothesis-tester, not a solver.** SunCalc (mourner/suncalc, BSD-licensed, zero dependencies, available via NPM and CDN) is *the* chronolocation tool Bellingcat references. The governing shadow relationship is `tan(sun_altitude) = object_height / shadow_length`, and the sun azimuth gives camera-facing direction. Critically, it does not solve for a location: you supply candidate location(s) and it returns sun azimuth/altitude plus shadow length for a given date and time so you can confirm or exclude — exactly the interaction Greyline's existing MapLibre place-picker already supports. The worked Lisbon example demonstrates shadow-direction and sun-azimuth chronolocation as a qualitative-plus-trigonometric method. (https://github.com/mourner/suncalc, https://www.bellingcat.com/resources/2020/12/03/using-the-sun-and-the-shadows-for-geolocation/)

**Bellingcat's OSM geolocation search is a simplified Overpass front-end that Greyline can extend directly.** Bellingcat ships a custom OSM geolocation search tool — a simplified Overpass query-language front-end that finds a location satisfying several visible features at once (for example, near a school AND a supermarket AND a sidewalk-street, within a maximum distance of each other). Greyline already has an Overpass connector (used for surveillance cameras on `/map`), so a "feature-cluster geolocation" query builder is a direct extension rather than new infrastructure. (https://bellingcat.gitbook.io/toolkit)

**Bellingcat's toolkit is living, staff-reviewed documentation that models methodological transparency.** Bellingcat's current toolkit was rebuilt in 2024 by staffer Johanna Wild during a Nieman-Berkman Klein fellowship at Harvard. It is community-maintained, staff-reviewed, living documentation organized by category (geolocation, maps and satellites, and others), and it stresses transparency about satellite imagery's accuracy and temporal-resolution limits. This is the model for how Greyline should present methodology — cite limitations and show your work — rather than producing black-box answers. (https://bellingcat.gitbook.io/toolkit)

### Verification methodology and chain-of-custody (SIFT, Tactical Tech)

**SIFT is the dominant, teachable verification framework, empirically grounded in lateral reading.** Caulfield's SIFT method (Stop; Investigate the source; Find better coverage; Trace claims, quotes, and media to original context) is the dominant, teachable verification framework. It is grounded in Wineburg and McGrew's finding that professional fact-checkers using lateral reading were roughly 2x better than PhD historians and roughly 5x better than students at judging source trustworthiness. SIFT maps cleanly onto Greyline's existing checklist component pattern (the OPSEC checklist, the hotel scorecard) as a guided per-claim verification flow. (https://spotlight.ebu.ch/p/sift-method-investigative-verification-guide)

**Tactical Tech's Exposing the Invisible Kit supplies an explicit chain-of-custody definition and a warning about third-party metadata.** The Kit frames every investigation around four ingredients — evidence, verification, documentation, and safety — and gives an explicit chain-of-custody definition: "a process that seeks to demonstrate that physical and other sorts of evidence is protected from tampering during the course of an investigation, from the point of collection to the point of publication or use in court." It also warns that metadata on photos you did not take "can be very easily altered and should not be relied on as evidence." Both points directly inform a local case-file with hash-based chain-of-custody. (https://kit.exposingtheinvisible.org/en/)

### Offline reverse-image and metadata tooling (perceptual hashing, EXIF)

**Offline reverse-image and near-duplicate detection is achievable entirely client-side with perceptual hashing.** blockhash-core is pure-JS with no large binary (ideal for Greyline's bundle), while phash-js needs a roughly 4MB ImageMagick WASM, and Meta's PDQ has Rust-to-WASM bindings for higher robustness. Hashes are 32-128 bits and compared by Hamming distance, robust to resize and recompress — enabling "have I seen this image before in my case?" and near-duplicate clustering without ever uploading the image. Actual web reverse-image *search* (Google Lens, Yandex) stays a copy-the-link hand-off, exactly like Greyline's self-doxxing tool already does. (https://github.com/freearhey/phash-js)

**Greyline's existing EXIF analyzer is already an OSINT-grade metadata viewer.** `server/services/exif.ts` already performs best-effort TIFF/EXIF parsing in pure TypeScript and surfaces GPS, device serials, MakerNote, embedded thumbnails (which can be pre-edit or uncropped versions), and lint severities. This is already comparable to Jeffrey's Image Metadata Viewer and is the natural anchor for a broader "verify this media" workflow that chains metadata, then perceptual hash, then chronolocation, then reverse-image hand-off.

### Feature notes and supporting rationale

- **Chronolocation / Shadow Geolocation Lab** — Shadow/sun analysis is the single most-cited Bellingcat chronolocation technique; it is hypothesis-testing (place candidate → check sun), which is exactly Greyline's existing map-pick interaction. Evidence: the Bellingcat toolkit SunCalc entry and the Lisbon shadow worked example.
- **Document Sanitizer** — Dangerzone is the canonical journalist tool for opening untrusted source documents and for true redaction; Tactical Tech warns that redaction-by-overlay leaks the underlying text. Greyline already strips EXIF, so extending to full-document neutering is the obvious next step. Evidence: dangerzone.rocks, FPF.
- **Investigation Case File with Chain-of-Custody** — Tactical Tech defines chain-of-custody as protecting evidence from tampering from collection to publication or court; ICIJ/Datashare workflows revolve around a durable case index. Greyline's vault and SQLite repositories already provide the storage and crypto primitives. Evidence: Exposing the Invisible Kit; Datashare.
- **Local Entity Extraction & Cross-Reference** — NER plus cross-referencing is the core of Datashare and Aleph (Aleph's standout is cross-dataset relationship mapping). Greyline already generates copy-ready OSINT queries, so closing the loop document → entity → query is natural. Evidence: ICIJ Datashare FAQ; Aleph entity-extraction docs.
- **Offline Reverse-Image & Near-Duplicate Detector** — Detecting recycled/old images repurposed as "breaking" is a core verification step (SIFT's "trace to original"); doing the comparison locally avoids leaking sensitive evidence to a search engine. Evidence: blockhash/phash-js libraries; First Draft/SIFT verification guidance.
- **SIFT Verification & Source-Protection Playbooks** — SIFT is the dominant teachable verification framework (fact-checkers 2-5x better than experts via lateral reading); journalists need an explainer for source intake they cannot self-host. Greyline's strength is curated checklists (54-item OPSEC, hotel scorecard). Evidence: Caulfield SIFT; CJR SecureDrop guide; EFF June-2025 border checklist.
- **OSM Feature-Cluster Geolocation Search** — Bellingcat built exactly this as a "simplified Overpass" to find geolocation leads from co-located features; Greyline already ships an Overpass connector (surveillance cameras) and MapLibre, so this is incremental. Evidence: Bellingcat OSM search toolkit entry.

## Open questions

- **Positioning vs persona.** How much OSINT/journalism surface should a "gray man personal-security" product expose before it confuses positioning? Is the journalism pillar a separate "mode"/persona, or features woven into the existing surveillance/maps/travel pillars?
- **NER bundle-size budget.** For local NER, what bundle-size budget is acceptable? compromise/wink-nlp (tens of KB, weaker) vs a transformers.js ONNX model (tens of MB, lazy-loaded) — and does shipping a model conflict with the "tiny, fast, offline-first" promise?
- **Sanitizer security guarantee.** Does true in-browser document sanitization (pdf.js pixel-rebuild) provide a strong-enough security guarantee without a real sandbox like Dangerzone's gVisor, or should Greyline be explicit that it neuters content but is not a malware air-gap?
- **Chain-of-custody signing.** Should chain-of-custody exports be cryptographically signed (and with what key — the vault's Argon2id-derived key, or a per-case key), and is that needed for civilian use cases or only when positioning toward actual journalists?
- **Legal/liability surface.** Shipping OSINT/self-doxxing/people-search hand-offs and "find a location from a photo" tooling — does the AGPL local-only model adequately limit Greyline's exposure, and should features carry an explicit ethics/lawful-use gate?
- **Offline spatial index.** Which offline geocoding/spatial index to use for the OSM feature-cluster search when running against a bundled Geofabrik extract rather than the live Overpass connector?
