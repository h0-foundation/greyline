# Maps / Geospatial Intelligence: Leveling Up Greyline's Map Pillar

_Part of the Greyline research corpus. Generated from cited 2026 web research; see Sources._

_Offline-first vector tiles, local routing/isochrones, viewshed, and OSM safety layers — every component self-hostable, air-gappable, and AGPL/MIT/BSD-compatible._

## Executive summary

Greyline's map today is a single online raster basemap (CARTO dark, MapLibre GL v5.18) with a handful of opt-in overlays. The `pmtiles@4.4.0` package is already a dependency but is never instantiated, and there is no vector tiling, terrain, routing, isochrones, or viewshed anywhere in the pipeline. This is the single largest gap between Greyline's "offline-first, privacy-first" marketing and what the `/map` route actually delivers: the map is the one core surface that still requires a live network connection and leaks tile requests to a third party (CARTO).

The 2023-2026 open-source geospatial landscape closes every one of those gaps with local-only, self-hostable components that fit Greyline's architecture exactly. **Offline vector tiles** are now a solved problem via Protomaps + PMTiles: the full planet is a single ~120 GB `.pmtiles` file (z0-15, OSM/ODbL data, BSD style code), and `pmtiles extract --bbox ... --maxzoom N` pulls a city- or country-scoped slice (Philadelphia is a few MB; a whole-planet z0-6 overview is ~60 MB) that serves directly from `/public` over HTTP Range requests with zero server. **Local routing and isochrones** are covered by Valhalla (MIT), whose `/isochrone` endpoint returns GeoJSON time/distance contours — a turnkey "how far can I get in N minutes" egress engine that runs offline on regional extracts.

On top of that, **advanced cartography** is now native: MapLibre v5 ships globe projection and a standalone hillshade layer driven by `raster-dem` sources, and Mapterhorn provides free Terrarium-encoded DEM tiles as PMTiles ("Protomaps for terrain") for both hillshade and client-side viewshed. **GEOINT primitives** — SunCalc for sun/shadow chronolocation, turf.js for bearing/dispersion/great-circle math, and deck.gl for GPU density heatmaps — all run client-side with no network. Finally, **OSM safety layers** (`lit=*` street lighting, `man_made=surveillance` with directional cones and ALPR plate readers, `amenity=hospital[emergency=yes]`, `amenity=police`) are bundleable into the same region pack via the existing Overpass client or a one-time Geofabrik extract.

The net result is a credible path to truly air-gapped maps with real roads, labels, terrain, routing, and tradecraft overlays — a capability Google, Apple, and Mapbox structurally cannot offer to a privacy product sold to lawyers, journalists, and aid workers. The work decomposes into one XL foundation (tile packs) and a series of S/M/L features layered on top, most of which reuse dependencies Greyline already has or that are tiny BSD/MIT additions.

## Key findings

### Offline vector tiles: the foundation is one unused dependency away

**Greyline already imports `pmtiles@4.4.0` but never instantiates it.** `components/map/map-view.tsx` uses only a raster CARTO source; the entire offline-vector-tile pillar is one unused dependency away from being wired up. The MapLibre protocol registration is two lines: `import {Protocol} from 'pmtiles'; maplibregl.addProtocol('pmtiles', new Protocol().tile)` followed by a source `url: 'pmtiles:///geo/region.pmtiles'`. Once the `.pmtiles` file is in `/public`, there is no runtime network and no server — MapLibre reads tiles via HTTP Range requests against the static file.

**The Protomaps full-planet basemap is a single ~120 GB `.pmtiles` file** (z0-15, ODbL data + BSD style code). The command `pmtiles extract <planet> region.pmtiles --bbox=W,S,E,N --maxzoom N` produces an offline regional slice with no server. Concrete sizes from the Protomaps docs: a Philadelphia extract is a few MB, US+Mexico at z0-15 is ~17 GB, and a z0-6 whole-planet overview is only ~60 MB. Each additional zoom level roughly doubles the size, so trip-scoped packs stay small as long as max zoom is bounded to what a city walk-route needs (z14-15). Daily planet builds are published at `maps.protomaps.com/builds`. Source: https://docs.protomaps.com/basemaps/downloads

**OpenFreeMap's Liberty style is a drop-in dark/editorial vector style** for the Protomaps PMTiles (BSD code / CC-BY design), and it is actively maintained — unlike the abandoned Positron/Bright/Dark/Fiord OpenMapTiles styles. It matches Greyline's Field Atlas/Dossier aesthetic. Important caveat: OpenFreeMap itself does **not** serve PMTiles (it serves MBTiles/Btrfs, and Cloudflare range-request latency was the cited reason), so Greyline must self-bundle PMTiles from the Protomaps planet build rather than pointing at OpenFreeMap as a tile host. Source: https://openfreemap.org/

### Local routing and isochrones: Valhalla is the clear pick

**Valhalla is the only one of Valhalla / OSRM / GraphHopper with native isochrones plus a tiled graph designed for offline/low-memory devices and regional extracts.** It is MIT-licensed, ~5.8k stars, v3.7.0 (released 2026-04-29), and written in C++. Its local `/isochrone` API on `localhost:8002` returns GeoJSON polygons for up to 4 time- **or** distance-contours per costing mode (pedestrian / auto / bicycle / multimodal) — directly usable as an egress/extraction "reachability" overlay answering "how far can I get in N minutes." Tesla ships Valhalla in-car, which validates the embeddable/constrained-hardware story. Sources: https://github.com/valhalla/valhalla and https://valhalla.github.io/valhalla/api/isochrone/api-reference/

**The competitive trade-offs ground the Valhalla choice.** Per the GIS-OPS practitioner overview, the rule of thumb is "OSRM for matrices, Valhalla for dynamic/low-spec, GraphHopper for lower-spec servers." OSRM is faster for distance matrices but has **no isochrones** and needs 64-300 GB RAM, which is incompatible with a laptop-local install. GraphHopper supports isochrones but is a heavier Java embed sitting in the middle. For Greyline's offline, low-footprint, isochrone-centric use case, Valhalla's tiled hierarchical graph (which can be built from a regional Geofabrik extract) is the only fit. Source: https://gis-ops.com/open-source-routing-engines-and-algorithms-an-overview/

### Terrain, hillshade, and globe: native in MapLibre v5

**MapLibre GL v5.0 (January 2025) added globe projection and v5.24 (June 2025) added a standalone hillshade layer** with multidirectional illumination — both driven by `type: 'raster-dem'` sources Greyline does not yet use. The globe projection auto-falls back to Mercator around z12 to preserve float32 precision. A known caveat: hillshade tiles do **not** rotate when draped on 3D terrain, so hillshade-on-terrain combinations need care. Source: https://maplibre.org/maplibre-gl-js/docs/examples/

**Mapterhorn (2025) is "Protomaps for terrain."** Led by ex-MapLibre coordinator Oliver Wipfli and NLnet-funded, it distributes free Terrarium-encoded WebP 512px DEM tiles as PMTiles at `download.mapterhorn.com/planet.pmtiles`. It is bbox-extractable and mergeable across zoom bands (`pmtiles merge`), giving Greyline an offline elevation bundle usable for **both** hillshade rendering and viewshed/line-of-sight compute. The keyless online fallback is AWS `elevation-tiles-prod` Terrarium tiles, decoded as `elevation = (R*256 + G + B/256) - 32768`. Source: https://protomaps.com/blog/mapterhorn-terrain/

### Client-side GEOINT: viewshed, sun/shadow, and dispersion math

**Viewshed / line-of-sight is computable client-side from Terrarium DEM** via the classic radial angle-accumulation algorithm: a target point is visible iff the elevation angle to it exceeds the running maximum angle along that radial. Earth-curvature correction matters beyond a few kilometers. There is no actively-maintained pure-JS library (thingless/viewshed is Python; RFAnalysisJS is stale), so the path is a ~150-line port of the GRASS `r.viewshed` / Izraelevitz-Xdraw approach over decoded DEM tiles, run in a Web Worker. Source: https://en.wikipedia.org/wiki/Viewshed_analysis

**SunCalc computes sun azimuth/altitude fully offline** (BSD, by `mourner`, based on Jean Meeus algorithms, matching the NOAA SPA within ~0.01 deg across +/-80 deg latitude). Shadow direction = `azimuth + 180 deg`; shadow length = `height / tan(altitude)`. This powers both photo-geolocation/chronolocation verification (a journalism workflow, and a named technique in Bellingcat's toolkit) and golden-hour / low-light egress timing. Source: https://github.com/mourner/suncalc

**turf.js v7.3 provides the rest of the client-side geospatial math** (modular, MIT): `bearing`, `destination` (Haversine), `greatCircle`, and `isolines`. Greyline can build dispersion fans, choke-point geometry, and great-circle flight legs with zero network and no new heavy dependency. Source: https://turfjs.org/docs/api/greatCircle

**deck.gl HeatmapLayer is the GPU path for density overlays.** It integrates with MapLibre via `MapboxOverlay({interleaved: true})` + `map.addControl(overlay)` (requires `maplibre-gl > 3`, satisfied by v5.18) and can sit below labels via `beforeId`. This renders CCTV/incident-density heatmaps without re-rendering thousands of point markers. The trade-off (see Open questions) is bundle weight versus MapLibre's built-in heatmap layer. Source: https://deck.gl/docs/developer-guide/base-maps/using-with-maplibre

### OSM safety layers: bundleable, directional, offline

**OSM safety layers are fully bundleable offline.** `lit=*` on ways (`lit=yes/no/24-7/automatic`) is the correct tag for street lighting — not just `highway=street_lamp` nodes. `man_made=surveillance` carries `camera:direction` (cardinal, or degrees clockwise from north), `camera:type` (fixed/dome/panning), `camera:mount`, and `surveillance:type=ALPR` for license-plate readers — enough metadata to draw directional coverage cones and distinguish plate readers. `amenity=hospital[emergency=yes]` and `amenity=police` provide extraction anchors. All of this is fetchable via the existing Overpass client or a one-time Geofabrik extract baked into the region PMTiles pack. Sources: https://wiki.openstreetmap.org/wiki/Tag:man_made=surveillance and https://wiki.openstreetmap.org/wiki/Overpass_turbo/Examples

**Greyline's Overpass client currently queries only `man_made=surveillance`.** `server/api-clients/overpass.ts` discards the directional tags and fetches nothing else. A single Overpass union query (`nwr`, `around:`, `{{bbox}}`) can add `lit` ways, hospitals, police, and ALPR cameras in the same request. Separately, the `saved_routes` table (migration 003) is schema-only with no UI, making it the natural persistence home for SDR (surveillance-detection routes) and E&E (escape-and-evasion) routes generated by Valhalla + turf.

### Sequencing notes

The **Offline Regional Tile Packs** feature (XL) is the foundation — it instantiates the dormant `pmtiles` dependency, replaces the network-only CARTO raster, and produces the region pack into which every other layer (safety OSM, terrain DEM) is baked. The two **S** items (Sun & Shadow, Camera Coverage Cones) are cheap, high-signal upgrades that reuse SunCalc/turf and the existing Overpass client and can ship independently before the tile-pack work lands. The Valhalla-dependent items (Egress Isochrone Planner L, SDR/E&E Route Layer M) share the same local routing sidecar and should be planned together. The **Viewshed Exposure Analyzer** (L) depends on the Mapterhorn DEM slice already being in the region pack, so it follows the tile-pack foundation.

## Open questions

- **Valhalla packaging.** Should Valhalla ship as a bundled Docker sidecar (heavier install, true turnkey local routing) or as an optional connector the user self-hosts? The former strengthens "it just works offline" but conflicts with the lean `pnpm dev` / single-container story.
- **Tile-pack size budget per trip.** Is a sub-GB country-level z0-12 pack acceptable for the target user's disk, or should packs be city-scoped (few-MB) with on-demand zoom? This drives whether viewshed/hillshade DEM ships in every pack or only on request.
- **Viewshed scope for v1.** Is a custom Web-Worker viewshed port (no maintained JS lib) within scope, or should v1 ship only 2D camera-coverage cones and great-circle/bearing GEOINT and defer terrain viewshed to a later milestone?
- **Safety-layer baking pipeline.** How should the app bake OSM safety layers (lit/CCTV/hospitals/police) into PMTiles offline — via a one-time Geofabrik PBF -> tilemaker/planetiler pipeline at pack-build time, or keep them as a separate small GeoJSON overlay fetched once via Overpass and cached? The former is fully air-gapped; the latter is simpler but needs one online fetch.
- **ODbL vs AGPL interaction.** Does shipping OSM-derived data inside the app (ODbL share-alike) interact with Greyline's AGPL-3.0 in a way that needs explicit attribution/licensing notes in `THIRD_PARTY_LICENSES.md` for each bundled tile pack?
- **deck.gl bundle cost.** deck.gl is a sizable dependency (~hundreds of KB) — is the GPU heatmap worth it over MapLibre's built-in heatmap layer, given Greyline's editorial/lightweight ethos and the React 19 bundle budget?
