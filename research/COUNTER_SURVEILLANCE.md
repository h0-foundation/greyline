# Counter-Surveillance & Surveillance-Detection: Science + Offline-First Tooling for Greyline

_Part of the Greyline research corpus. Generated from cited 2026 web research; see Sources._

## Executive summary

Greyline already carries the structural foundations of a serious counter-surveillance module, but ships almost none of the surface area. The `saved_routes` table (`server/db/migrations/003_routes.sql`) declares a `CHECK(type IN ('sdr','extraction','variation','normal'))` constraint and carries origin/destination/waypoints/geometry/distance/duration columns — a fully-specified Surveillance Detection Route store with zero UI wired to it. The `counter_surveillance_log` table is backed by a `repeatMatches()` heuristic in `server/db/repositories/field.ts` that only counts recurring free-text descriptors and ignores the Time/Environment/Distance axes that define the tradecraft. An Overpass connector already fetches `man_made=surveillance` camera nodes, and a MapLibre map already supports opt-in OSINT layers. The missing pieces map cleanly onto a focused, sellable product rather than a research project.

The underlying tradecraft is well-codified and partially algorithmically assistable. TEDD — Time, Environment, Distance, Demeanor — is the canonical correlation test for distinguishing coincidence from hostile surveillance, and a Surveillance Detection Route (SDR) is a scripted path that forces a tail through choke points to reveal correlation without alerting the watchers (Stratfor/TorchStone lineage). The keystone implementable precedent is OSRM-CCTV (arXiv:2108.09369), the first CCTV-aware router: it weights OpenStreetMap edges using camera radius/view-angle/direction tags to offer a privacy mode (avoid cameras) and a safety mode (prefer cameras). The same machinery, inverted and extended, yields an SDR planner that routes deliberately *through* choke points and observation-rich segments. DeFlock (deflock.org, ~50k of 100k+ US Flock ALPRs mapped into OSM, EFF-backed against a Flock cease-and-desist) and the OSM `surveillance` schema make the camera/ALPR layer reachable today through Greyline's existing Overpass proxy.

The strongest privacy argument for a self-audit feature is de Montjoye et al., "Unique in the Crowd" (Sci Rep 2013): four spatio-temporal points uniquely identify 95% of 1.5M individuals, and uniqueness decays only as roughly the 1/10 power of resolution. Greyline's own trip log is therefore a re-identifiable pattern-of-life that the app can score and let the user redact. For physical TSCM, the honest civilian truth is sobering: peer-reviewed hidden-camera detection (LAPD/SenSys 2021 at 88.9% vs 46% naked-eye; CamRadar/UbiComp 2023; HeatDeCam/CCS 2022) all require ToF/RF/thermal sensors a browser cannot touch. Greyline should ship a guided sweep protocol plus a flashlight/lens-glint visual aid, not fake automated detection. Tracker-scanning rides the Apple/Google DULT standard (IETF `draft-ietf-dult-accessory-protocol`, service UUID `0xFCB2`, near-owner bit, 24h MAC rotation when separated), but Web Bluetooth cannot passively scan advertisements — so this is a companion-concept with an AirGuard/native handoff. Drone awareness is feasible only as an OpenDroneID handoff (ASTM F3411, BLE/WiFi-NAN).

Net: an SDR planner, a TEDD-scored log, a pattern-of-life self-audit, and an OSM choke-point/CCTV layer are the high-utility, genuinely offline-first wins. The TSCM/tracker/drone surfaces are valuable only as honest, handoff-aware guidance.

## Key findings

### Latent schema and dead-code: what Greyline already half-built

**The `saved_routes` table is a complete SDR store with no UI.** `server/db/migrations/003_routes.sql` already declares `CHECK(type IN ('sdr','extraction','variation','normal'))` alongside origin/destination/waypoints/geometry/distance/duration columns. It is a fully-specified Surveillance Detection Route store with zero UI wired to it. This is the single highest-leverage gap to close: the schema was clearly designed for SDR planning and is dead weight until built.

**The existing TEDD detector ignores TEDD.** `repeatMatches()` in `server/db/repositories/field.ts` only tallies recurring `person_desc`/`vehicle_desc` strings (count > 1). It never checks whether the repeats span different Times, different Environments, or meaningful Distances — which the tradecraft literature (Stratfor/TorchStone, <https://www.torchstoneglobal.com/detecting-hostile-surveillance/>) says is the entire point. A person seen twice at the same cafe is noise; the same person across town hours later is the signal. The lat/lng and timestamp columns required to score this properly are already stored and unused.

**The Overpass camera connector is raw-points-only and timeout-prone.** `server/api-clients/overpass.ts` queries only `man_made=surveillance`, returns raw points, times out above zoom 8 (the world-bbox is too large and the layer too sparse), and ignores ALPRs. Critically, `cameraPopup()` already parses `camera:direction` and `surveillance:zone` — the data needed to draw approximate coverage cones is already in hand. A CCTV-density heatmap needs server-side aggregation/caching (the `api_cache` table already exists).

**Greyline ships no routing engine and no street-level OSM data.** `package.json` contains only `maplibre-gl` and `pmtiles`, both client-side render libraries. The `data/` directory holds country JSON, and `public/geo` carries only `countries-110m.geojson` (the scratch map). An offline SDR planner therefore needs a client-side router (OSRM-WASM, or a bundled A*/Valhalla per-city tiles) over an opt-in fetched OSM extract — never a hosted OSRM call.

### The implementable precedent: OSRM-CCTV

**OSRM-CCTV is the keystone implementable precedent for surveillance-aware routing.** OSRM-CCTV (arXiv:2108.09369, <https://arxiv.org/abs/2108.09369>) ingests OSM `man_made=surveillance` nodes with camera radius, view-angle, and direction; re-weights the routing graph; and offers a privacy mode (avoid cameras) and a safety mode (prefer cameras). It was released open-source. Inverting and extending its edge-reweighting gives an SDR planner that routes *through* choke points and high-observation segments rather than away from them — turning a privacy router into a detection router.

**Choke points have a documented dual nature.** Per TorchStone, the same narrow passage that flushes out a tail is also an ideal ambush "perch" site. An SDR planner must therefore surface choke points as *both* detection tools and risk markers, and must recommend varying routes and times to avoid becoming predictable — the core pattern-of-life vulnerability. A planner that only treats choke points as a benefit would be tradecraft-naive.

### The privacy science: your own log is the threat surface

**Four points uniquely identify 95% of people.** de Montjoye et al., "Unique in the Crowd: The privacy bounds of human mobility" (Nature Scientific Reports 2013, srep01376, <https://www.nature.com/articles/srep01376>), shows that four spatio-temporal points uniquely identify 95% of 1.5M individuals, and that uniqueness decays only as roughly the 1/10 power of spatial/temporal resolution — i.e., coarsening your data barely helps. This is the citable basis for a pattern-of-life self-audit that scores Greyline's own trip log.

**Dataset scale does not save you.** The country-scale follow-up, "The risk of re-identification remains high even in country-scale location datasets" (Patterns / Cell Press 2021, <https://www.sciencedirect.com/science/article/pii/S2666389921000143>), shows unicity stays around 0.93 even at 1M-person scale with only 3 auxiliary points. This refutes the "safety in numbers" intuition and strengthens the self-audit rationale: being one of a million does not anonymize a distinctive movement pattern.

### Honest limits: TSCM, BLE trackers, and drones in a browser

**Browser-based TSCM is fundamentally limited.** Peer-reviewed hidden-camera detection all requires hardware a web app cannot access: LAPD (ACM SenSys 2021, NUS/Yonsei, <https://dl.acm.org/doi/10.1145/3485730.3485941>) achieves 88.9% detection vs 46.0% naked-eye using a smartphone Time-of-Flight laser glint; CamRadar (UbiComp 2023) uses RF; HeatDeCam (ACM CCS 2022) uses thermal. None of ToF/RF/thermal is reachable from a browser. The realistic offline feature is a guided sweep checklist plus a camera-flashlight lens-glint / IR-on-front-camera visual aid — explicitly **not** automated detection. The literature itself warns against the false-confidence "detector" apps that dominate app stores.

**BLE tracker scanning cannot be done honestly in-browser.** The Apple/Google DULT standard (IETF `draft-ietf-dult-accessory-protocol`, <https://datatracker.ietf.org/doc/draft-ietf-dult-accessory-protocol/>) defines what a tracker scan would look for: BLE service-data UUID `0xFCB2`, non-owner GATT `15190001-12F4-C226-88ED-2AC5579F2A85`, a near-owner bit, and 24h MAC rotation in the separated state. This is detected at the OS level, and Web Bluetooth offers no passive advertisement scan. The correct Greyline move is to educate and hand off to native: OS-built-in alerts have shipped on Android and iOS since May 2024 (<https://www.apple.com/newsroom/2024/05/apple-and-google-deliver-support-for-unwanted-tracking-alerts-in-ios-and-android/>), with AirGuard / Tracker Detect as the open-source fallback.

**Drone awareness is a native-handoff problem.** Android throttles WiFi/BLE scans, and only flagship phones (Galaxy S10/S20) reliably receive Bluetooth Long-Range / WiFi-NAN OpenDroneID — so even a native drone-awareness feature is constrained. For Greyline (web) it can only be an OpenDroneID OSM app handoff plus an ASTM F3411 explainer, never in-app reception (see `opendroneid/receiver-android`, <https://github.com/opendroneid/receiver-android>).

### Free and near-free data layers

**DeFlock makes ALPR data a near-zero-cost layer.** DeFlock (<https://deflock.org/>) crowdsources Flock Safety ALPR cameras into OpenStreetMap as `man_made=surveillance` + `surveillance=ALPR` — ~50k of 100k+ US cameras mapped, EFF-backed against Flock's cease-and-desist. Because it lives in OSM, Greyline's *existing* Overpass connector already reaches ALPRs simply by adding `surveillance=ALPR` to the query. The SF Standard (Mar 2026) built a per-route camera-count tool on this exact data, validating the use case.

**The OSM `surveillance` schema already matches Greyline's parser.** The OSM `Key:surveillance` schema (<https://wiki.openstreetmap.org/wiki/Key:surveillance>) covers camera/guard/ALPR/gunshot_detector plus `camera:direction` (degrees from North) and `surveillance:zone` — exactly the tags `cameraPopup()` already parses. This enables coverage-cone estimation and a CCTV-density layer with no new parser. The "Surveillance under Surveillance" (sunders) project renders this hourly.

### The academic gap: do not over-claim ML detection

**No peer-reviewed algorithm formalizes human tailing-detection.** There is no published algorithm that models human tailing-detection as spatiotemporal-coincidence anomaly detection. The nearest transferable building blocks are trajectory-anomaly work — LM-TAD (SIGSPATIAL 2024, <https://arxiv.org/abs/2409.15366>), which treats trajectories as token sequences and uses perplexity/surprisal to flag anomalies, and context-aware TAD (2024) — plus co-moving BLE-tracker detection (AirCatch's "persistence + core-density" criterion). These target detours and fraud, not human tails. Greyline can ship a defensible TEDD heuristic but should **not** over-claim ML detection. AirGuard (TU Darmstadt SEEMOO, <https://www.researchgate.net/publication/360628055_AirGuard_-_Protecting_Android_Users_from_Stalking_Attacks_by_Apple_Find_My_Devices>) is the academic + shipping reference for the co-movement detection that is feasible — on a native platform, not in a browser.

### How Greyline wires its dormant `saved_routes` table

The SDR planner is the feature that activates the dormant schema. `003_routes.sql` already enumerates `type IN ('sdr','extraction','variation','normal')` and carries `geometry`, `waypoints`, `distance_m`, and `duration_s` — the columns an SDR planner needs to persist a route. The wiring is therefore: (1) a MapLibre drawing surface that captures origin, destination, and intended meeting point; (2) a local routing pass (OSRM-WASM or a hand-rolled A*/contraction-hierarchy router over an opt-in, cached, per-city OSM extract — Greyline ships no street graph today, so this extract is the prerequisite); (3) choke-point scoring derived from OSM street-graph topology (betweenness / medial-axis narrowness / single-exit bottleneck detection) and `man_made=surveillance` density via the existing Overpass connector; (4) edge re-weighting modeled on OSRM-CCTV but inverted to *prefer* choke points and high-observation segments; (5) a `variation` generator that emits N divergent routes; and (6) persistence into `saved_routes` with `type='sdr'`/`'variation'`/`'extraction'`. Every generated route must annotate choke points as both detection opportunities and ambush-perch risks, and must nudge the user to vary route and timing to avoid the pattern-of-life predictability the self-audit feature quantifies.

## Open questions

- **Offline routing engine choice for the SDR planner:** bundle OSRM-WASM (large, heavy binary), ship Valhalla tiles per-trip-city, or hand-roll a client-side A*/contraction-hierarchy over an opt-in-fetched OSM extract? Each has very different bundle-size vs coverage tradeoffs given Greyline ships no street data today.
- **How to source the street-level OSM graph offline-first without a hosted service:** fetch + cache a bounding-box extract per trip destination at trip-creation time (opt-in connector), or require the user to drop in a Geofabrik/Protomaps extract? This determines whether the SDR planner works truly offline.
- **Choke-point scoring metric on the OSM street graph:** edge betweenness centrality, Voronoi/medial-axis narrowness, or single-exit/topological bottleneck detection — which gives tradecraft-meaningful choke points without false positives, and how to validate it against real SDR practice?
- **Legal exposure of an SDR/camera-avoidance planner:** some jurisdictions forbid mapping/alerting on traffic-enforcement and security infrastructure (OSM wiki note). Does Greyline need per-region feature gating or disclaimers for the CCTV/ALPR layer and SDR routing?
- **Self-audit data scope:** should the pattern-of-life unicity scoring use only the user's logged trips, or also offer an optional ingest of phone Location History / GPX for a fuller (but riskier-to-store) picture — and how to keep that strictly local + vault-encrypted?
- **TEDD "Environment" axis:** is a bundled coarse place-type classifier acceptable, or does it need the opt-in Nominatim connector (a network dependency that breaks the pure-offline promise for that one signal)?
- **Overpass caching aggressiveness:** how to cache/aggregate camera data server-side without violating the offline-first/opt-in model — is a per-trip-city camera extract bundled at planning time acceptable, and how stale can it be before it misleads (DeFlock/OSM camera data changes fast)?
